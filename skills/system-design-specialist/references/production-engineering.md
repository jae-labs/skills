# Reliability, Observability & SLO/SLI Engineering

A guide to planning operational resilience, observing distributed systems, and setting quantitative health metrics.

---

## 1. Reliability & The Hostile Path (NALSD Mindset)

Production environments are hostile. A professional system design must explicitly specify how the system behaves during failures:

### A. Cascading Outage Prevention
- **The Problem:** One service slows down, causing upstream services to run out of connection threads, taking down the entire app.
- **Mitigation:**
  - **Circuit Breaker:** Wrap downstream requests. If downstream error rate exceeds a threshold (e.g., 5% over 10 seconds), trip the breaker and fail fast with cached/default data.
  - **Timeout:** Never wait indefinitely. Set a strict timeout (e.g., 200ms) on all downstream calls.
  - **Load Shedding:** If a service's queue depth exceeds capacity, immediately drop incoming low-priority requests (HTTP 503 / 429) to protect the process core.
  - **Bulkhead Isolation:** Partition resources (thread pools, connection pools) per downstream dependency. If the payment service pool is exhausted, the user profile pool remains unaffected. Prevents a single slow dependency from consuming all shared resources.

### B. Disaster Recovery (DR)
- **RPO (Recovery Point Objective):** The maximum amount of data loss acceptable (e.g., 1 hour = hourly snapshot replication).
- **RTO (Recovery Time Objective):** The maximum time allowed to restore service (e.g., 15 minutes = requires automated DNS failover and multi-AZ standby).

### C. Multi-Region Active-Active
- **The Problem:** Syncing databases across the globe without hitting write latency limits (CA to Singapore = 150ms latency).
- **Mitigations:**
  - Route user traffic to the nearest geographic region (GeoDNS).
  - Use eventual consistency for regional writes.
  - Employ conflict resolution strategies: Last-Write-Wins (LWW), Application-level merges, or Conflict-Free Replicated Data Types (CRDTs).

### D. Client Retries, Exponential Backoff, and Jitter
- **The Problem**: When a service experiences an outage or transient slowdown, clients that retry failed requests immediately can cause a "retry storm" (thundering herd), crushing the backend and preventing it from recovering.
- **Exponential Backoff**: Clients must increase the delay between retries exponentially (e.g., 1s, 2s, 4s, 8s...) to give downstream services recovery time.
- **Jitter (Randomness)**: To prevent all clients from retrying in lockstep (synchronized traffic spikes), add random noise (jitter) to the backoff duration.
- **Formula**:
  $$\text{Delay} = \text{random}(0, \min(\text{cap}, \text{base} \cdot 2^{\text{attempt}}))$$
- **Mutations & Idempotency**: All write/mutation retries must pass a unique `idempotency_key` (generated by the client) in the HTTP headers. The API Gateway caches processed key responses in Redis for 24 hours to prevent duplicate transactions (e.g., duplicate credit card charges).

### E. Distributed Transactions (Consistency vs. Availability)
When scaling databases via sharding or transitioning to microservices, single-node ACID transactions no longer work. You must coordinate consistency:
- **Two-Phase Commit (2PC)**: A centralized coordinator prepares all database nodes (Phase 1) and commits them only if all agree (Phase 2).
  - *Trade-off*: Strong consistency (ACID-compliant), but blocking protocol (locks held during phase 2 reduce throughput) and high latency overhead.
- **Saga Pattern (Compensating Transactions)**: A series of local transactions commit immediately. If a step fails, the system executes compensating (rollback) transactions backwards.
  - *Trade-off*: Eventual consistency, but highly scalable and non-blocking. Implement via **Orchestration** (central coordinator) or **Choreography** (decentralized events).

### F. Leader Election & Distributed Coordination
When a distributed system needs a single coordinator (e.g., one node writes to a shard, one scheduler triggers jobs):
- **Raft:** Nodes elect a leader through majority vote. Leader handles all writes and replicates to followers. Used by etcd, CockroachDB, Consul.
- **Paxos:** More general consensus protocol. Harder to implement correctly. Used by Google Chubby, Spanner.
- **Lease-based:** A node acquires a time-limited lease (via ZooKeeper, etcd) to act as leader. Must renew before expiry or another node takes over.
- **When to mention:** Any time your design has a "single coordinator" or "only one writer" — explain how that leader is elected and what happens during leader failure.

### G. SPOF Elimination, Redundancy, & Failover Systems
Designing a fault-tolerant architecture requires eliminating any Single Point of Failure (SPOF):
- **Redundancy Models:**
  - **Active-Passive (Hot Standby):** Secondary node replicates state from the active node. Upon failure, traffic is rerouted to the standby. Introduces a brief failover window where service might be degraded or offline.
  - **Active-Active:** Multiple nodes process requests concurrently. If one node fails, load balancers automatically redistribute the traffic to remaining healthy nodes. Eliminates failover latency but requires stateless application layers.
  - **N+1 / 2N Redundancy:** Deploying more instances than the minimum required (N) to handle peak capacity, ensuring the system can tolerate one or more node failures.
- **Failover Mechanisms:**
  - **L4/L7 Health Checking:** Load balancers continuously monitor instance health. Unhealthy nodes must be removed within seconds to prevent request blackholing.
  - **DNS-Based Failover:** Routes traffic across multiple endpoints. *Caveat:* High DNS TTL values can delay failover propagation up to several hours due to ISP caching. Keep TTL low (e.g. 10–60s) for high-availability endpoints.

### H. Stateful Replication Models & Engineering Impact
Data must be replicated across independent storage nodes for durability and availability. Selecting a replication model dictates the system's runtime performance and consistency guarantees:
- **Single-Leader Replication:**
  - *Mechanism:* All writes are sent to the leader node. Leader writes locally and sends data change stream to followers. Reads can be served by any node.
  - *Synchronous:* Leader waits for confirmation from at least one follower before acknowledging the write.
    - *Impact:* Guarantees zero data loss (low RPO) and strong read consistency. However, a single replica failure blocks all write operations, reducing availability.
  - *Asynchronous:* Leader acknowledges writes immediately. Replicas catch up in the background.
    - *Impact:* Low write latency and high availability. However, introduces **Replica Lag**, resulting in eventual consistency anomalies (e.g. read-your-writes violation, stale reads). Data written to the leader but not yet replicated will be lost if the leader crashes (high RPO risk).
- **Multi-Leader Replication (Active-Active Database):**
  - *Mechanism:* Writes can be processed by any regional leader node and are replicated cross-region.
  - *Impact:* Extremely low write latency for global users and survives full data center outages. However, introduces write conflicts (e.g., concurrent updates to the same row). Requires resolving conflicts using Last-Write-Wins (LWW), Conflict-Free Replicated Data Types (CRDTs), or application-level reconciliation.
- **Leaderless Replication:**
  - *Mechanism:* Clients write and read from multiple database nodes directly.
  - *Impact:* High tolerance for node failures. Consistency is tuned using Read (R) and Write (W) Quorums over total replica count (N). Setting $W + R > N$ guarantees that reads overlap with writes, providing strong consistency, but increases write amplification and network traffic.

## 2. Observability Engineering

### A. Monitoring vs Observability
- **Monitoring** answers pre-defined questions: "Is CPU above 80%?" "Is error rate above 1%?" These are **known-unknowns** — you must decide what to watch before failures happen.
- **Observability** lets you ask arbitrary questions of your system *after* a problem occurs, without deploying new code. It answers **unknown-unknowns** — "Why are 2% of users in Brazil seeing 3s latency on Tuesday afternoons?"
- **Implication for design:** Observability requires emitting rich, high-cardinality, structured telemetry data (not just aggregate metrics). Every request should carry enough context to reconstruct its full journey.

### B. The Three Pillars
**Metrics** — Aggregated numerical measurements:
- **RED Method (for Services):** Rate (QPS), Errors (5xx/timeouts per second), Duration (P50, P90, P99 histograms).
- **USE Method (for Infrastructure):** Utilization (% busy), Saturation (queue depth), Errors (hardware faults).

**Logs** — Discrete event records:
- Never use plain text logs in production. Use **Structured JSON Logging**.
- Every log entry must include: `trace_id`, `span_id`, `service_name`, `timestamp`, `level`, and request-specific fields.

**Traces** — End-to-end request paths:
- Use **Distributed Tracing** (OpenTelemetry, Jaeger) to visualize execution across service boundaries. Essential for diagnosing tail latency (P99/P99.9) and failure correlation in microservice chains.
- **Trace Context Propagation Mechanics:** To maintain a single trace across distributed systems, trace identifiers must be injected into request metadata by upstream services and extracted by downstream services.
  - **W3C Trace Context (Standard):** Employs the `traceparent` header (format: `version-trace_id-parent_id-trace_flags`, e.g., `00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01`) and the optional `tracestate` header (for vendor-specific metadata). This is the default industry standard.
  - **B3 Propagation:** Employs multiple headers (`X-B3-TraceId`, `X-B3-SpanId`, `X-B3-Sampled`) or a single combined `b3` header. Often used in older Zipkin-based systems.
  - **Baggage:** Key-value pairs (e.g., `tenant_id=enterprise_alpha`) carried in the `baggage` header. Propagated globally down the request tree without changing service interfaces, allowing downstream systems to slice logs or traces by business context.
- **Asynchronous Tracing Context Propagation:**
  - *Message Queues (Kafka, RabbitMQ, ActiveMQ):* Trace context must not be lost when requests transition from synchronous HTTP/gRPC to asynchronous event processing. Upstream publisher services inject `traceparent` and `baggage` into message/record metadata (e.g., Kafka Record Headers, RabbitMQ message properties). The downstream consumer service extracts these headers from the message, sets them as the parent context of its execution thread, and creates a child consumer span (e.g. `messaging.receive`). This links the asynchronous worker execution directly to the original client request trace.
  - *In-Process Thread/Task Hand-offs:* When delegating work to background thread pools or async routines, pass the context explicitly (e.g. Go's `context.Context`) or wrap executors (e.g. Java's `ExecutorService`) to copy trace thread-locals to worker threads, preventing trace fragmentation.

### C. Wide Events & High-Cardinality Data
Traditional metrics aggregate data (e.g., "average latency = 50ms"), destroying the detail needed to debug outliers. **Wide events** (or structured events) are rich, per-request records containing dozens of fields:
- user_id, endpoint, status_code, latency_ms, cache_hit, db_query_count, region, build_version, feature_flags, error_message...
- These enable slicing and dicing production data along any dimension after the fact, without predefined dashboards.
- **Tools:** Honeycomb, Grafana Tempo, Datadog APM.

### D. Observability-Driven Development (ODD / Instrumentation-First)
Observability is a **design-time concern**, not an operations afterthought:
- **Telemetry Specifications:** Design span hierarchies, naming conventions (following OpenTelemetry Semantic Conventions), log structures, and metric dimensions during the API design phase.
- **Instrument Entry and Exit Points:** Instrument all ingress points (controllers, queue consumers), egress points (HTTP clients, database callers, queue publishers), and database/cache transactions from day 1.
- **Inject Business Context:** Bind business identifiers (`order_id`, `driver_id`, `payment_provider`) as span attributes. This allows querying traces for specific customer escalations.
- **Telemetry Verification (Testing):** Verify trace context propagation in integration tests. Assert that the Trace ID matches across the API Gateway, upstream services, message queues, and downstream workers. If a trace boundary is broken, fail the build.
- **Symptom vs. Cause Metrics Collection:**
  - *Symptom Metrics (Service Level):* Collect RED metrics (Rate, Errors, Duration) on all interfaces.
  - *Cause Metrics (Infrastructure Level):* Collect USE metrics (Utilization, Saturation, Errors) on memory, database CPU, disk IOPS, and network bandwidth.
  - *Custom Business Metrics:* Collect metrics that track transactional states (e.g., matching engine queue size, active matching rate, cache hit/miss ratio for driver locations) to detect functional degradation that infrastructure metrics might miss.

## 3. SLO & SLI Engineering

Do not state arbitrary SLAs. Formulate real SLOs and SLIs:

- **SLI (Service Level Indicator):** A quantifiable metric measuring real-time performance.
  - *Example:* "Percentage of successful, non-error HTTP GET requests to `/api/v1/user` completed in under 200ms."
- **SLO (Service Level Objective):** The target percentage for the SLI.
  - *Example:* "$\ge 99.9\%$ of requests satisfy the SLI over a rolling 30-day window."
- **SLA (Service Level Agreement):** The legal commitment to the customer, incorporating consequences/credits.
  - *Example:* "If availability falls below 99.9% in a month, refund 10% of subscription fees."
  - **Rule of Thumb:** Make the SLO tighter than the SLA to give operations team an error budget cushion.

### SLO-Based Alerting (Burn Rate)
Static threshold alerts ("CPU > 80%") generate noise and miss real user impact. SLO-based alerting measures **how fast you are consuming your error budget:**
- **Error Budget:** If your SLO is 99.9% over 30 days, you can tolerate 0.1% errors = ~43 minutes of downtime/month.
- **Burn Rate:** How fast the budget is being consumed. A burn rate of 1× means you'll exhaust the budget exactly at the end of the window. A burn rate of 10× means you'll exhaust it in 3 days.
- **Alert policy:** Page on-call at **14.4× burn rate** (budget exhausted in ~2 days) with a short window (1 hour). Ticket at **3× burn rate** (budget exhausted in ~10 days) with a longer window (6 hours).
- **Error Budget Policy:** When the error budget is exhausted, freeze feature deployments and redirect engineering effort to reliability work until the budget recovers.

### Symptom-Based vs Cause-Based Alerting
- **Symptom-based (preferred):** Alert on user-facing impact — "P99 latency exceeds SLO", "error rate above budget burn rate." Directly tied to user pain.
- **Cause-based (secondary):** Alert on infrastructure signals — "CPU > 90%", "disk > 85%." Useful for capacity planning but generates false positives. Use as supplementary context, not primary pages.

## 3b. Chaos Engineering & Incident Response

### Chaos Engineering
Validate that your observability and resilience patterns actually work under failure:
- **Principle:** Proactively inject failures in production (or staging) to discover weaknesses before real outages expose them.
- **Techniques:** Kill random service instances, inject network latency/packet loss, simulate AZ/region failures, exhaust connection pools, corrupt cache entries.
- **Tools:** Chaos Monkey (Netflix), Litmus, Gremlin, AWS Fault Injection Simulator.
- **Prerequisite:** Your observability must be strong enough to detect and diagnose injected failures. If chaos experiments cause undetected impact, fix observability first.

### Incident Response & Blameless Postmortems
- **Incident Commander:** Designate a single person to coordinate response, communicate status, and delegate investigation.
- **Blameless Postmortem:** After every significant incident, write a structured document: timeline, root cause, impact, detection time, resolution, and **action items** (not blame). Focus on systemic improvements.
- **MTTD / MTTR:** Track Mean Time to Detect (how fast you notice) and Mean Time to Resolve (how fast you fix). Improving MTTD is an observability problem; improving MTTR is a runbook/automation problem.

## 4. Security, Access Control & Data Residency

### A. Access Control: RBAC vs ABAC

System architectures must implement security boundaries to enforce authorization:

- **Role-Based Access Control (RBAC)**: Simplifies access by grouping permissions into roles (e.g., `admin`, `billing_editor`, `viewer`). Users are mapped to roles.
  - *Scale Implementation*: User roles are baked as cryptographically verified claims inside client JWTs. The API Gateway decodes the JWT and passes the role downstream in the header (`X-User-Role: billing_editor`).
- **Attribute-Based Access Control (ABAC)**: Dynamically grants access by evaluating attributes (e.g., *User department = Finance*, *Resource type = PaymentInfo*, *Environment IP = 10.x.x.x*, *Time = BusinessHours*).
  - *Scale Implementation*: Use a centralized Policy Decision Point (PDP) service (like Open Policy Agent - OPA) sidecar co-located with microservices to check attributes locally without adding network hop latency.

### B. Data Residency & Sovereignty
Data residency regulations (like GDPR in Europe, FedRAMP in the US, or HIPAA for healthcare data) mandate that citizen data must be stored and processed strictly within geographic boundaries.
- **Geographic Database Partitioning (Geo-Sharding)**: Shard data keys by geographic location. For example, all rows matching `region: EU` are hosted on EU-located database clusters, and `region: US` on US clusters.
- **Replication Boundary Fences**: Restrict database replica replication pools. EU read replicas must remain physically in the EU. Global active-active synchronization pipelines must strip PII or encrypt sensitive columns before copying cross-border.
- **Shared-Nothing Regional Deployments**: Fully isolate regional environments (e.g., `eu-west-1` and `us-east-1`). Use independent frontend, gateway, and database instances. Only non-PII events (like billing aggregates or audit logs) are piped back to global analytics warehouses.

### C. Distributed Authentication & Token Validation at Scale
Querying a central session database on every request introduces latency bottlenecks. Secure access is managed using:
- **Stateful Sessions**: Session IDs stored in a central Redis cache. Safe and easy to revoke instantly, but requires a cache lookup on every request.
- **Stateless JWTs**: Cryptographically signed tokens (JWT) containing user claims and TTL. Zero DB hits to verify, but difficult to revoke before expiry. (Mitigate using a local **Bloom Filter** gate and a **Redis Blacklist** for revoked tokens).
- **Identity Propagation**: The API Gateway authenticates the client JWT once at the edge, extracts claims, and injects them as downstream HTTP headers (e.g., `X-User-Id`), letting downstream microservices trust the identity.

### D. TLS Encryption, Termination & mTLS
Managing cryptographic handshakes consumes substantial CPU. Layer encryption to optimize:
- **TLS Termination**: The edge Load Balancer decrypts HTTPS traffic and routes it internally as plain HTTP over a secure private VPC network. Offloads CPU load and unifies certificate management.
- **TLS Pass-Through**: Routes raw encrypted TCP to backend app servers. Secure end-to-end, but high CPU load on app servers, and prevents the Load Balancer from making L7 HTTP routing decisions.
- **mTLS (Mutual TLS)**: Secure internal service-to-service communications using mutual certificate authentication, typically managed transparently via sidecar proxies (e.g., Envoy in a Service Mesh like Istio).

## 5. Operational Readiness Checklist

Ensure the system has:
- [ ] **Liveness Probe**: Processes are alive and deadlocks are detected (simple ping).
- [ ] **Readiness Probe**: Services are fully initialized and dependencies are reachable before receiving traffic.
- [ ] **Graceful Shutdown**: SIGTERM signals catch traffic stop, finish pending requests, and close db connections before exiting.
- [ ] **Data Residency Compliance**: Geo-sharded keys and replication boundaries configured to satisfy regional constraints.
- [ ] **Access Control Verification**: API Gateway validates JWT claims and propagates `X-User-Role` headers downstream.
- [ ] **Runbooks**: Documentation for critical on-call alert mitigations.
